# Codemagic CD Setup

This skill is **tool-agnostic**: different AI coding environments (Cursor, Claude Code, Gemini CLI, etc.) expose different tools. Follow the workflow using whatever capabilities your environment provides.

## Overview

This skill interactively generates a `codemagic.yaml` configuration file for Flutter projects. It asks the user a series of questions to determine their project structure, flavors, signing requirements, and publishing targets, then generates a fully configured Codemagic pipeline.

## Workflow

### Step 1: Gather Project Information

Ask the user the following questions **one at a time**, waiting for each answer before proceeding. Use your environment's way to prompt the user (e.g., `AskUserQuestion` in Claude Code or equivalent).

**1.1 — App name and working directory**

- "What is the name of your app? (e.g., `My App`)"
- "What is the working directory for the app relative to the repo root? (e.g., `.` for root, `apps/my_app` for a mono-repo)"

**1.2 — Flavors / Environments**

- "Which flavors (build environments) does your app use? List them separated by commas, or say 'none' if you don't use flavors. Common examples: `development, staging, production`"
- For each flavor, ask: "What is the entrypoint file for the **{flavor}** flavor? (e.g., `lib/main_{flavor}.dart`)"
  - If they said 'none', the entrypoint defaults to `lib/main.dart`.

**1.3 — Platform targets**

- "Which platforms are you targeting? (android, ios, or both)"

**1.4 — iOS Configuration** (skip if not targeting iOS)

- "What is the iOS bundle identifier? (e.g., `com.example.myapp`)"
- "Do you have an App Store Connect API key configured in Codemagic? (yes/no)"
  - If yes: "What is the integration name? (default: `iOS - App Store Connect API Key`)"
- "What iOS distribution type do you need? (`app_store`, `ad_hoc`, or `development`)"

**1.5 — Android Configuration** (skip if not targeting Android)

- "What is the Android application ID? (e.g., `com.example.myapp`)"
- "Do you have Android code signing configured in Codemagic? (yes/no)"
  - If yes: "What is the Android signing configuration name? (e.g., `Android Release Keystore`)"

**1.6 — Triggering**

- For each flavor/workflow, ask:
  - "What should trigger the **{flavor}** workflow?"
    - Options: `push to branch`, `tag`, `pull request`, or `manual only`
  - If push/PR: "Which branch pattern? (e.g., `main`, `develop`, `release/*`)"
  - If tag: "Which tag pattern? (e.g., `v*`)"

**1.7 — Publishing**

- "Do you want to publish iOS builds to TestFlight? (yes/no)"
- "Do you want to distribute Android builds via Firebase App Distribution? (yes/no)"
  - If yes: "What Firebase credential group name is configured in Codemagic? (e.g., `firebase_credentials`)"
- "Do you want email notifications for build results? (yes/no)"
  - If yes: For each flavor, ask: "What email addresses should receive **{flavor}** build notifications? (comma-separated)"

**1.8 — Build configuration**

- "What is the maximum build duration in minutes? (default: 60)"
- "Which Flutter channel do you want to use? (`stable`, `beta`, or a specific version like `3.24.0`)"
- "Which Java version? (default: 17)"
- "Which Xcode version? (`latest` or a specific version like `15.4`)" (skip if not targeting iOS)
- "For production Android builds, do you want to build an AAB (App Bundle) or APK? (default: AAB)"

### Step 2: Detect Existing Configuration

Before generating, check if a `codemagic.yaml` already exists in the repo root:

1. Search for `codemagic.yaml` in the repository.
2. If found, warn the user: "A `codemagic.yaml` already exists. Do you want to overwrite it or create a new file (e.g., `codemagic.yaml.new`)?"

### Step 3: Generate codemagic.yaml

Generate the `codemagic.yaml` file using the gathered information. Follow this structure:

yaml

# Codemagic CI/CD for {App Name}

# Generated by VGV AI Assistant

definitions:
environment: # Email notification lists (one per flavor/workflow)
{flavor}\_emails: &{flavor}\_emails - {emails}

workflows:

# Per-flavor workflow

{flavor}:
name: {Flavor Display Name}
max_build_duration: {max_duration}
working_directory: {working_directory}

    # iOS App Store Connect integration (if applicable)
    integrations:
      app_store_connect: {integration_name}

    environment:
      flutter: {flutter_channel}
      java: {java_version}
      xcode: {xcode_version}
      cocoapods: default
      groups:
        - {credential_groups}
      vars:
        BUNDLE_ID: {ios_bundle_id}
        FLAVOR: {flavor}
        ENTRYPOINT: {entrypoint}
      android_signing:
        - {android_signing_name}
      ios_signing:
        distribution_type: {distribution_type}
        bundle_identifier: $BUNDLE_ID

    triggering:
      events:
        - {trigger_event}
      branch_patterns:  # or tag_patterns
        - pattern: {pattern}
          include: true
      cancel_previous_builds: true

    cache:
      cache_paths:
        - $HOME/Library/Caches/CocoaPods
        - $HOME/.pub-cache
        - $HOME/.gradle/caches

    scripts:
      - name: Get Flutter packages
        script: flutter pub get

      # iOS-only steps
      - name: Set up code signing (iOS)
        script: xcode-project use-profiles

      - name: Install CocoaPods
        script: find . -name "Podfile" -execdir pod install \;

      # Android build
      - name: Build Android {APK or AAB}
        script: flutter build {apk|appbundle} --release --flavor $FLAVOR -t $ENTRYPOINT --build-number=$PROJECT_BUILD_NUMBER

      # iOS build
      - name: Build iOS IPA
        script: flutter build ipa --release --flavor $FLAVOR -t $ENTRYPOINT --build-number=$PROJECT_BUILD_NUMBER --export-options-plist=/Users/builder/export_options.plist

    artifacts:
      - build/**/outputs/**/*.apk
      - build/**/outputs/**/*.aab  # only for AAB builds
      - build/**/outputs/**/mapping.txt
      - build/ios/ipa/*.ipa
      - /tmp/xcodebuild_logs/*.log

    publishing:
      app_store_connect:  # if TestFlight enabled
        auth: integration
        submit_to_testflight: true
      firebase:  # if Firebase App Distribution enabled
        firebase_service_account: $FIREBASE_SERVICE_ACCOUNT
        android:
          app_id: $FIREBASE_APP_ID_ANDROID_{FLAVOR}
          groups:
            - internal-{flavor}
          artifact_type: apk
      email:
        recipients: *{flavor}_emails

#### Generation Rules

1. **No flavors**: If the user has no flavors, generate a single workflow named after the app. Omit `--flavor` and `-t` flags from build commands. Use `lib/main.dart` as entrypoint.

2. **Platform-specific steps**: Only include iOS steps (signing, CocoaPods, IPA build) if targeting iOS. Only include Android steps if targeting Android.

3. **Production AAB vs APK**: Use `flutter build appbundle` for production Android builds if the user chose AAB. Use `flutter build apk` for non-production or if user chose APK.

4. **Artifacts**: Only include artifact patterns for the platforms being built (e.g., no `.ipa` patterns if iOS is not targeted).

5. **Publishing**: Only include publishing sections the user opted into. Omit `app_store_connect` if not publishing to TestFlight. Omit `firebase` if not using Firebase App Distribution. Omit `email` if no notifications.

6. **Triggering**: If `manual only`, omit the `triggering` section entirely for that workflow.

7. **Environment groups**: Only include `groups` if the user has Firebase or other credential groups.

8. **Signing**: Omit `android_signing` if not configured. Omit `ios_signing` if not targeting iOS.

9. **YAML anchors**: Use YAML anchors (`&name` / `*name`) for email lists to keep the file DRY when there are multiple workflows.

### Step 4: Review with User

After generating the file:

1. Show a summary of what was generated (number of workflows, platforms, triggers).
2. Ask: "Does this look correct? Would you like to adjust anything?"
3. Apply any requested changes.

### Step 5: Write the File

Write the final `codemagic.yaml` to the repository root (or the user's chosen location).

## Important Notes

- Always use `$PROJECT_BUILD_NUMBER` for build numbers — this is Codemagic's auto-incrementing build number.
- The `export-options-plist` path `/Users/builder/export_options.plist` is Codemagic's default path for iOS export options.
- `xcode-project use-profiles` is Codemagic's CLI tool for iOS code signing setup.
- CocoaPods installation uses `find` to handle mono-repo structures where the Podfile may be nested.
- Cache paths should always include CocoaPods, pub cache, and Gradle caches for optimal build times.
